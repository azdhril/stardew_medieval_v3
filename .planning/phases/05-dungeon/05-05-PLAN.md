  <files>src/UI/DialogueBox.cs, src/Scenes/DialogueScene.cs</files>
  <action>
Fix the hardcoded 960x540 overlay so dialogue renders correctly at any viewport size (including fullscreen).

**In `src/UI/DialogueBox.cs`:**
1. REMOVE the `ScreenWidth`, `ScreenHeight`, `PanelX`, `PanelY` constants (they are viewport-dependent, not constants).
2. KEEP `PanelWidth = 880`, `PanelHeight = 120`, `PortraitSize = 80`, `Padding = 8` (these are fixed panel-internal sizes — do NOT scale with viewport; the panel is a fixed-size element anchored to the bottom-center of whatever viewport it lives in).
3. EXTEND the `Draw` signature to accept viewport dimensions as the LAST TWO parameters (additive, so existing indentation of earlier args is preserved):
   ```csharp
   public void Draw(SpriteBatch sb, SpriteFont font, Texture2D pixel,
       Texture2D? portrait, string revealedText, bool showAdvance, bool pulseOn,
       int viewportWidth, int viewportHeight)
   ```
4. At the top of `Draw`, compute runtime locals:
   ```csharp
   int panelX = (viewportWidth - PanelWidth) / 2;
   int panelY = viewportHeight - 32 - PanelHeight;   // 32px bottom margin, same offset as before
   ```
5. Replace the `new Rectangle(0, 0, ScreenWidth, ScreenHeight)` dim overlay with `new Rectangle(0, 0, viewportWidth, viewportHeight)` so the dim covers the entire real viewport.
6. Replace every other use of `PanelX` / `PanelY` in the method body with the runtime `panelX` / `panelY` locals (panel outline, panel fill, inner bevel DrawRectOutline call, portrait origin, text column origin, advance indicator position). The values inside those rectangles (PanelWidth/PanelHeight/PortraitSize/Padding offsets) do NOT change.
7. Update the class-level XML doc comment: change "the 880x120 dialogue panel anchored bottom-center" to "the 880x120 dialogue panel anchored bottom-center of the current viewport" to reflect the runtime binding.

**In `src/Scenes/DialogueScene.cs`:**
1. In `Draw(SpriteBatch spriteBatch)`, immediately before the `spriteBatch.Begin(...)` call, read the viewport:
   ```csharp
   var vp = Services.GraphicsDevice.Viewport;
   ```
2. Pass the viewport dimensions to `_box.Draw`:
   ```csharp
   _box.Draw(spriteBatch, _font, _pixel, _npc.Portrait, revealed, showAdvance, pulseOn,
       vp.Width, vp.Height);
   ```
3. Do not change any other code in DialogueScene — the state machine, typewriter, input handling, and SpriteBatch.Begin/End remain untouched.

**Why this approach (not alternatives):**
- Giving `DialogueBox` a `GraphicsDevice` field would leak a MonoGame dependency into a currently-stateless renderer and complicate unit testing. Passing the viewport by value keeps `DialogueBox` pure.
- Scaling the panel itself (e.g. making PanelWidth a function of viewportWidth) would change the "look and feel" the user already approves of in windowed mode. The fix is purely about positioning, not sizing.

**Gap reason being closed:** DialogueBox.cs uses hardcoded ScreenWidth=960, ScreenHeight=540 constants. Overlay draws a fixed 960x540 rectangle regardless of actual viewport (05-UAT.md Gap 1).

   ```csharp
   // === Gold label (under stamina bar, matches bar column) ===
   int gold = _inventory?.Gold ?? 0;
   string goldText = $"Gold: {gold}";
   int goldY = nextBarY + (staDrawn ? ScaledBgHeight(StaminaBarScale) : fallbackH) + barSpacing + 2;

   spriteBatch.DrawString(_font, goldText, new Vector2(barX + 1, goldY + 1), Color.Black);
   spriteBatch.DrawString(_font, goldText, new Vector2(barX, goldY), Color.Gold);
   ```
   This keeps the label in the same left-aligned column as the HP/MP/STA bars and uses the local `nextBarY` / `fallbackH` / `barSpacing` / `staDrawn` variables already in scope.
