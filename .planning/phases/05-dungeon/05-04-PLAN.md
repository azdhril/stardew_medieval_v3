## Why this strategy (TMX data fix, not tileset swap)

The debug session diagnosed GID=1 everywhere, but `dungeon_r1.tmx` has since
been manually patched to add a second tileset reference (`farm_tileset.tsx`
at firstgid=4481) and fill its Ground layer with GID=4554 (= localId 73 of
`0_Complete_Tileset_16x16.png`). That is the working reference pattern: it
uses the farm tileset's visible stone/dirt-ish tile as a dungeon floor. The
other six rooms were NOT updated and still fill Ground with GID=1, which
maps to the empty top-left corner of the props sheet -> black.

We are NOT creating a new dungeon-specific floor tileset (scope creep, no
asset for it). We are NOT swapping `dungeon_tileset.tsx` to a different
image (semantically fine but riskier: dungeon_tileset is also expected to
host wall/prop tiles in future plans). We ARE mirroring r1's fix across
the remaining six rooms: add the `farm_tileset.tsx firstgid="4481"` tileset
declaration alongside the existing `dungeon_tileset.tsx firstgid="1"` and
rewrite the Ground CSV to all-GID-4554.

If, on visual inspection, GID 4554 looks out of place for a dungeon (too
farm-green), the executor MAY pick a different localId in farm_tileset
that reads as stone/dirt and use GID = 4481 + chosen_localId. The decision
must be applied uniformly across all six rooms so the dungeon is visually
consistent. Whatever floor GID the executor picks MUST match what
`dungeon_r1.tmx` uses (update r1 too if picking a different tile) so the
aesthetic is uniform across the run.

## Defensive hardening (Task 2)

`Game1.Draw` currently calls `GraphicsDevice.Clear(Color.Black)`. If any
future map regresses to all-transparent tiles, the result is pitch black
again. Changing the clear color to a dark grey (e.g. `Color(24, 24, 28)`)
makes the regression visible rather than invisible. Scope: 1-line change
in `Game1.cs:90`. No behavior change in day/night overlay (it composites
alpha-black over whatever is beneath).
