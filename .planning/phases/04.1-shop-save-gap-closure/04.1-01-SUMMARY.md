# Phase 04.1 Plan 01: Remove Follow-Selection Scroll Behavior — Summary

One-liner: Deleted the `UpdateLayoutCache` block that forced `_scrollOffset` to track `_selectedIndex`, making shop scroll wheel-driven only per CONTEXT D-01/D-02.

## What Changed

Removed 4 lines from `src/UI/ShopPanel.cs` `UpdateLayoutCache()` (formerly lines 218-221):

```csharp
// DELETED:
if (_selectedIndex < _scrollOffset) _scrollOffset = _selectedIndex;
else if (_selectedIndex >= _scrollOffset + VisibleRows) _scrollOffset = _selectedIndex - VisibleRows + 1;
if (_scrollOffset < 0) _scrollOffset = 0;
```

Replaced with 2 comment lines citing D-01/D-02 rationale. The range-clamp above it (lines 213-216) remains the sole clamp, and the wheel handler at lines 92-98 is untouched.

## Why

The deleted block re-centered the scroll on `_selectedIndex` every frame inside `UpdateLayoutCache` — which Update() calls every tick. That both fought user wheel input (rubber-band on scroll) and contradicted the new Stardew-style UX (D-01: no persistent selection). Gating the block would preserve dead weight; deleting it closes the path entirely.

## Scroll Path After Change

- **Write site 1 (wheel handler, ShopPanel.cs ~93-98):** `_scrollOffset = Math.Clamp(_scrollOffset - Math.Sign(wheel), 0, maxScroll);`
- **Write site 2 (range clamp in UpdateLayoutCache, line 215):** `if (_scrollOffset > maxScroll) _scrollOffset = maxScroll;`
- **Write site 3 (scrollbar drag path, line 399):** pre-existing, untouched.

No path now reads `_selectedIndex` to write `_scrollOffset`.
