<objective>
Migrate all text rendering from MonoGame's baked `SpriteFont` (via Content Pipeline) to runtime TTF rasterization via `FontStashSharp`. This fixes the blurry `scale=1.35f` title regression in Shop and Chest UIs and unlocks correct rendering of Portuguese accented characters for free.

Purpose: Eliminate bilinear-scaled title distortion (visible bug in ShopPanel.cs and ChestScene.cs) and decouple UI text rendering from the MGCB content pipeline. Requesting native-size glyphs produces crisp output with no asset re-baking required for any future size change.

- New `FontService` wrapper at `src/Core/FontService.cs` owning two `FontSystem` instances (body + bold).
- `ServiceContainer.Fonts` slot wired in `GameplayScene.LoadContent`.
- All 18 call-site files migrated from `SpriteFont` to `SpriteFontBase`.

- Build passes; Shop/Chest titles render crisp at native bold size 16.

  <action>
    Create a new file `src/Core/FontService.cs` implementing the `FontService` class and `FontRole` enum as specified in RESEARCH.md (section "Recommended FontService wrapper").

    Required shape:

    ```csharp
    using System;
    using System.IO;
    using FontStashSharp;

    namespace stardew_medieval_v3.Core;

    /// <summary>Typeface role used to select which FontSystem to pull a glyph size from.</summary>
    public enum FontRole { Body, Bold, Small }

    /// <summary>
    /// Runtime TTF font cache. Owns one FontSystem per typeface family; each family
    /// lazily rasterizes any requested size and shares a single atlas across sizes.
    /// </summary>
    public sealed class FontService : IDisposable
    {
        private readonly FontSystem _body;
        private readonly FontSystem _bold;

        public FontService(string fontDir)
        {
            var regularPath = Path.Combine(fontDir, "NotoSerif-Regular.ttf");
            var boldPath    = Path.Combine(fontDir, "NotoSerif-Bold.ttf");
            Console.WriteLine($"[FontService] Loading fonts from: {Path.GetFullPath(fontDir)}");

            _body = new FontSystem();
            _body.AddFont(File.ReadAllBytes(regularPath));

            _bold = new FontSystem();
            _bold.AddFont(File.ReadAllBytes(boldPath));
        }

        /// <summary>Returns a SpriteFontBase cached internally by the underlying FontSystem.</summary>
        public SpriteFontBase GetFont(FontRole role, int size) => role switch
        {
            FontRole.Bold  => _bold.GetFont(size),
            FontRole.Small => _body.GetFont(size),
            _              => _body.GetFont(size),
        };

        public void Dispose()
        {
            _body.Dispose();
            _bold.Dispose();
        }
    }
    ```

    Pitfalls to avoid (per RESEARCH.md):
    - Do NOT `new FontSystem()` more than once per typeface family — all sizes share one atlas.
    - Do NOT call this constructor from a Draw hot path — it does disk I/O.
    - DO log the resolved absolute path once at construction so Pitfall 2 (path resolution) is diagnosable on first run.

    The `FontRole.Small` case deliberately maps to the same `_body` FontSystem — there is no separate "small" TTF; the old NotoSerifSmall was just NotoSerif baked at 10pt. The enum value exists for call-site clarity only.
